# Деплой на платформу Zerovibe

Публикует приложение: платформа собирает Docker-образ из ТВОЕГО кода (Dockerfile
проекта), запускает его в Kubernetes-кластере и выдаёт поддомен на домене приложений
`zerovibe.app` с авто-TLS. Локально Docker не нужен — сборка идёт на платформе.

> Адрес приложения берётся из поля `domain` в ответе на деплой (см. Шаг 3) — НЕ
> хардкодь домен сам. Сейчас это `<sub>.zerovibe.app`.

## Предусловия

- Ключ настроен (`~/.zerovibe/config.json`). Нет → `/zerovibe:install`.
- Папка привязана (`ZEROVIBE_PROJECT` в `.env`). Нет → `/zerovibe:init`.
- В корне есть `Dockerfile` (как в эталоне; приложение слушает `:8080`).
- Для локальной проверки/предпросмотра нужен Go (ставится при `/zerovibe:install`).
  Если его нет — предупреди, что без него не получится показать приложение до
  публикации, и предложи `/zerovibe:install`.

Креды и привязка: ключ и базовый `api` — из `~/.zerovibe/config.json`; `ZEROVIBE_PROJECT`
(и, если задан, переопределяющий `ZEROVIBE_API`) — из `.env` проекта.

## Шаг 0 — проверить и показать пользователю ДО публикации

Это важно: креатор должен увидеть, что приложение работает, ещё до прода.

1. **Проверь, что собирается и тесты зелёные** (в проекте есть `make check`). Покажи
   результат продуктовым языком: «проверил — всё работает» / «нашёл проблему, чиню».
   **Красное — НЕ публикуем**, сначала чини (см. skill `testing-rules`).

2. **Подними приложение локально для живого предпросмотра** и дай пользователю ссылку
   `http://localhost:8080`, чтобы он посмотрел вживую. Важные инварианты локалки:
   используй ВРЕМЕННУЮ базу (не трогай рабочие данные), `SECURE_COOKIE=false` и
   `APP_BASE_URL=http://localhost:8080` (локалка без TLS); точка входа — `cmd/server`;
   готовность проверяй через `/healthz`. Дождись подтверждения пользователя.

3. **Останови предпросмотр и убери временную БД** перед публикацией.

## Шаг 1 — собрать архив исходников

gzip-архив кода. КРИТИЧНО: `Dockerfile` и исходники лежат в КОРНЕ архива (без
обёртки-папки). Исключи: `.git`, `data`, `*.db`, `tmp`, `node_modules`, `.env`
(данные и секреты не заливаем).

## Шаг 2 — залить архив

> `POST {api}/dev/projects/{ZEROVIBE_PROJECT}/source` с `X-API-Key`.
> Ответ: `upload_url` (presigned PUT) и `key`.
> Залей архив `PUT`-ом на `upload_url`. `key` пойдёт как `source_key` в деплой.

## Шаг 3 — запросить деплой

> `POST {api}/dev/projects/{ZEROVIBE_PROJECT}/deploy` с `X-API-Key`, тело
> `{"subdomain": "<sub>", "source_key": "<key из шага 2>"}`.

Поддомен `<sub>`: только `a-z`, `0-9`, дефис; 1..63 символа; уникален на платформе.
Спроси у пользователя, если не задан.

Ответ содержит `domain` (полный адрес приложения, напр. `<sub>.zerovibe.app`),
`status` и (для первого деплоя приложения) `setup_url` + `setup_token`.

**Деплой асинхронный.** Сразу после запроса `status` обычно `pending` — это нормально:
платформа приняла задачу, образ собирается и раскатывается в кластере (десятки секунд —
пара минут). Открой `https://<domain>/healthz` и подожди, пока ответит `200` (готово),
или перепроверь статус через `GET {api}/dev/deployments` — там у развёртывания
появится `running`. Статусы:
- `pending` / `building` → ещё собирается/раскатывается, подожди и проверь `/healthz`.
- `running` → готово, открывай `https://<domain>` (проверь `/healthz` → 200).
- `failed` → причина: занятый поддомен, красные тесты или ошибка сборки образа.

## Шаг 4 — первичная настройка (создать первого админа)

В приложении ещё нет администратора — заведи его аккаунт **сам, запросом**. Пользователь
не программист и НЕ открывает никакие ссылки и не вводит коды — все технические действия
делаешь ты. Платформа вернула в ответе деплоя `setup_token` и `setup_url` (вида
`https://<domain>/setup`) — бери их ИЗ ОТВЕТА, это для тебя, не для показа человеку.

1. **Спроси у пользователя только продуктовое:** какой email и пароль сделать для входа
   владельца (пароль придумывает он). Больше ничего — про токен/URL не упоминай, это деталь.
2. **Отправь запрос сам:** `POST {setup_url}` с телом
   `{"email": "<его email>", "password": "<его пароль>", "token": "<setup_token из ответа деплоя>"}`.
   - `201` → готово, скажи продуктово: «завёл тебе вход, открывай приложение и заходи».
   - `410` → админ уже есть (настройка завершена ранее) — вход уже создан, просто скажи войти.
   - `403` → код не подошёл (редко: токен из другого деплоя) — передеплой и повтори с новым `setup_token`.

> НЕ давай человеку ссылку «зайди и заведи сам» и НЕ показывай ему `setup_token`/`setup_url`.
> Запрос /setup — твоя работа, человек только называет желаемые email и пароль.

## Что знать о платформе (грабли и факты)

- Все вызовы API платформы — заголовком **`X-API-Key`** (не Bearer). API платформы на
  `zerovibe.ru`; приложения — на `zerovibe.app` (разные домены, так и задумано).
- Приложение запускается в Kubernetes (под на ноде кластера), TLS — общий wildcard-серт
  `*.zerovibe.app`, выдаётся автоматически.
- Данные приложения (SQLite) — на сетевом диске (PVC), переживают пересоздание пода и
  переезд на другую ноду. Приложение работает в одном экземпляре (single-instance).
- **Платформенные переменные прокидываются автоматически:** при деплое платформа сама
  кладёт в контейнер `PLATFORM_API_URL`/`PLATFORM_API_KEY` (отправка писем —
  восстановление пароля), `APP_BASE_URL` (свой домен), `SECURE_COOKIE` и `SETUP_TOKEN`
  (код первичной настройки — для создания первого админа, см. Шаг 4). Креатора про них
  не спрашивай.
- Миграции goose применяются при старте пода сами — новые таблицы появятся автоматически.

Аргумент пользователя (поддомен): $ARGUMENTS
