# Правила тестов zerovibe

Минимально достаточное покрытие, обязательное для каждой фичи. Два уровня:

## 1. Unit usecase (на фейк-репозитории)

На каждый сервис в `usecase` — тест с in-memory реализацией порта (фейк), без БД
и сети. Проверяй:
- успешный путь (создание/чтение возвращает ожидаемое);
- валидацию (пустые/слишком длинные поля → `ErrValidation`, проверка `errors.As`);
- бизнес-правила (порядок, лимиты, "не найдено" → `ErrNotFound`).

Фейк-репозиторий — простая структура со слайсом и реализацией методов порта.

## 2. E2E транспорта (httptest на временной SQLite)

На каждый раздел — тест полного стека (`db → repo → usecase → web`) на БД во
`t.TempDir()` (авто-очистка). Проверяй:
- мутация возвращает HTML-ФРАГМЕНТ (без `<html>`, с нужным классом/содержимым);
- `GET` страницы возвращает полную страницу с данными;
- удаление убирает сущность;
- негатив: несуществующий id → 404.

## Что НЕ тестировать

- Геттеры/тривиальные мапперы без логики.
- Сам `platform/db` (очередь записи) — он покрыт эталоном, не дублируй.

## Запуск

```sh
go test ./...          # все тесты, быстрые, без сети
go vet ./...           # статанализ
```

Все тесты должны проходить ДО деплоя. Деплой с красными тестами запрещён.

## Принцип

Покрытие — не ради процентов, а ради уверенности: бизнес-логика (usecase) и
контракт HTTP+HTML (transport) проверены. Этого достаточно для приложения такого
класса; не раздувай тесты на инфраструктуру.
