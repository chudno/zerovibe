# Деплой на хостинг zerovibe

Разворачивает приложение в контейнере на наших VM, выдаёт поддомен
`<subdomain>.zerovibe.ru` с авто-TLS. Деплоем управляет оркестратор (его API).

## Перед деплоем — проверь готовность

```sh
go build ./... && go vet ./... && go test ./...
```
Если что-то красное — НЕ деплой, сначала почини (см. skill testing-rules).
Убедись, что есть `Dockerfile` в корне (multi-stage, distroless, без CGO).

## Запрос деплоя

Деплой инициируется вызовом API оркестратора. Нужны:
- `owner_id` — идентификатор вайбкодера (выдаётся платформой при регистрации);
- `name` — человекочитаемое имя приложения;
- `subdomain` — поддомен (только `a-z`, `0-9`, дефис; 1..63 символа).

```sh
curl -fsS -X POST "$ZEROVIBE_API/v1/apps" \
  -H "Authorization: Bearer $ZEROVIBE_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"owner_id":"<owner>","name":"<name>","subdomain":"<sub>"}'
```

Переменные окружения (выдаёт платформа):
- `ZEROVIBE_API` — базовый URL оркестратора;
- `ZEROVIBE_TOKEN` — токен вайбкодера.

## Результат

Ответ содержит `domain` и `status`:
- `status: running` — приложение поднято, открывай `https://<subdomain>.zerovibe.ru`;
- `status: failed` — смотри причину; частые: занятый поддомен, красные тесты, ошибка сборки.

Проверка после деплоя:
```sh
curl -fsS "https://<subdomain>.zerovibe.ru/healthz"   # ok
```

## Замечания

- TLS Caddy выпускает автоматически при первом запросе (нужен корректный DNS).
- Данные приложения (SQLite) хранятся в volume на VM в РФ (152-ФЗ) и переживают рестарт.
- Поддомен уникален в пределах платформы — выбери свободный.

Аргумент пользователя: $ARGUMENTS
